01

Memory Management — Background

Core Idea

Memory is a large array of bytes, each with its own address. The CPU fetches instructions from memory based on the program counter. Memory management ensures processes get the memory they need, are protected from each other, and that memory is used efficiently.

🛡️ Base & Limit Registers

Two hardware registers protect each process's memory space:

  • Base Register — holds the starting physical address of the process
  • Limit Register — holds the size of the process's address range
  • Every memory access is checked: Base ≤ address < Base + Limit
  • Access outside this range → fatal error (process killed)

⚡ Why Memory Management?

  • Allocate memory to programs on demand
  • Track used and free memory at all times
  • Protect each process from accessing another's memory
  • Support multiprogramming — run many programs simultaneously
  • Improve overall CPU utilization and system performance

💥 Fragmentation — The Core Problem

Internal Fragmentation

Wasted space inside an allocated block. Happens when OS gives a process slightly more memory than it needs.

Example
Process needs 18 KB → OS gives 20 KB → 2 KB wasted inside the block. That 2 KB can't be used by anyone else.
External Fragmentation

Enough total free memory exists, but it's scattered — no single contiguous block is large enough.

Example
Free: 10 KB + 5 KB + 8 KB = 23 KB total. Process needs 15 KB. No single 15 KB block → allocation fails!
02

Address Space & Address Translation

Logical Address (Virtual Address)

Generated by the CPU while a program runs. Each process has its own logical address space starting from 0. Not real — must be translated to physical address before use. Also called a virtual address.

Physical Address

The real location in RAM hardware. Generated by the MMU (Memory Management Unit). Limited by actual RAM size. User programs never see or touch physical addresses directly.

📦 Components of a Process Address Space

Key point: Heap and Stack grow toward each other. The OS places them at opposite ends to allow maximum expansion without immediate collision.

🎯 Dynamic (Hardware-Based) Relocation

Allows a program compiled to start at address 0 to be placed anywhere in physical memory.

Physical Address = Virtual Address + Base
  • Base Register = where process starts in RAM
  • Bounds Register = size of the process
  • If VA ≥ Bounds → Protection Fault → process killed
Worked Example
Process size = 4 KB, loaded at 16 KB

VA 0 → PA = 16384 + 0 = 16384 ✅
VA 1 KB → PA = 16384 + 1024 = 17408 ✅
VA 3000 → PA = 16384 + 3000 = 19384 ✅
VA 4400 → 4400 > 4096 → FAULT ❌
03

Paging

Definition

Paging is a fixed-size partition memory management technique. Physical memory is divided into frames; logical memory is divided into pages of the same size. Pages can be placed in any available frames — non-contiguous allocation — eliminating external fragmentation.

📄 Pages

Fixed-size chunks of logical (virtual) memory. When a process runs, the OS divides it into pages.

Page size defined by hardware. Typically power of 2 (512B – 1GB).

🖥️ Frames

Fixed-size chunks of physical memory (RAM). Each frame holds exactly one page.

Page size = Frame size (always!)

📋 Page Table

OS-maintained data structure mapping each Page Number → Frame Number.

One page table per process. Indexed by Virtual Page Number (VPN).

🔢 Logical Address Structure

Physical Address = Frame Number × Page Size + Offset
Bit Calculation Formula
If logical address space = 2m and page size = 2n bytes:
 → Page Number bits = m − n  |  Offset bits = n

Example: 64 KB logical space (216), page size 16 B (24) → Page# = 12 bits, Offset = 4 bits

🗂️ Page Table Entry (PTE) — What's Inside?

FieldPurposeWhen = 1 / Set
Frame NumberPhysical frame holding this pageAlways present
Valid/Present BitIs page currently in RAM?Page is loaded in RAM
Dirty Bit (D)Has page been modified since loading?Page was written to → must write to disk before eviction
Reference Bit (A)Was page recently accessed?Page was read/written recently → used by LRU algorithm
Protection Bits (R/W/X)Read / Write / Execute permissionsAccess allowed

Paging — Advantages & Disadvantages

✅ Advantages
  • Eliminates external fragmentation
  • Simple memory allocation algorithm
  • Easy to swap equal-sized pages/frames
  • Enables non-contiguous allocation
❌ Disadvantages
  • Internal fragmentation (last page may be partially used)
  • Page tables consume extra memory
  • Multi-level paging → extra memory accesses overhead

📉 Reducing Page Table Size

Solution 1
Bigger Pages

Increase page size → fewer pages → fewer entries.
Drawback: More internal fragmentation.

Solution 2
Hybrid Paging + Segmentation

Each segment (code, heap, stack) gets its own page table. Unused segments take no space.

Solution 3
Multi-Level Page Tables

Page directory → page tables → frames. Invalid regions waste zero memory. Used in modern OSes.

TLB — Translation Lookaside Buffer

Definition

The TLB is a small, fast hardware cache inside the CPU that stores recently used (VPN → Frame Number) mappings. It dramatically speeds up address translation by avoiding slow main-memory page table lookups.

Without TLB vs With TLB

❌ Without TLB
1

CPU generates virtual address

2

Access page table in RAM

Memory access #1

3

Access actual data

Memory access #2

Total = 2 memory accesses
✅ With TLB (Hit)
1

CPU generates virtual address

2

Check TLB — HIT!

Frame number found immediately

3

Access actual data

Only 1 memory access needed

Total = 1 memory access ⚡

Steps on TLB HIT

1

CPU generates logical address

2

Check TLB — page number FOUND

3

Frame number retrieved from TLB

4

Access physical memory directly

Fast! Only 1 RAM access.

Steps on TLB MISS

1

CPU generates logical address

2

Check TLB — NOT found

3

Access page table in RAM

Slow — extra memory access

4

Frame number retrieved

5

TLB updated with new entry

If TLB full → replacement policy

📊 TLB Hit Rate Example — Array Access

Array of 10 integers (4 bytes each). Virtual address = 100. Page size = 16 bytes. 4 integers per page.

a[1]06HITSame page as a[0], spatial locality

Why Performance Improves
Spatial Locality: Nearby array elements share the same page → only first access per page causes a miss.
Temporal Locality: If a loop repeats, same pages hit again → near 100% hit rate.
05

Virtual Memory

Definition

Virtual Memory is a technique where the OS provides each process with an illusion of having a very large, private memory, by using disk space as an extension of RAM. A program can run even if its full size exceeds available physical memory.

The Apartment Analogy 🏢

  • Building = Physical Memory (RAM)
  • Each flat = A process
  • Every flat has the same layout (addresses start at 0)
  • Flat 101's kitchen and Flat 201's kitchen exist in different physical locations
  • This is exactly how virtual memory works — same virtual addresses, different physical locations
Study Table Analogy
Table = RAM (fast but limited)
Bookshelf = Hard Disk (large but slow)
Books = Program data

You can study 10 books even if your table fits only 3. Swap books to/from shelf as needed.

🎯 Three Goals of Virtual Memory

👻
Transparency

Program behaves as if it has its own private memory. OS creates the illusion behind the scenes.

⚡
Efficiency

Uses hardware (TLB) to minimize slowdown. Should use minimal extra memory overhead.

🛡️
Protection

Processes cannot access each other's memory. Prevents crashes and security breaches.

🔄 Swapping

Definition

A process is temporarily moved from RAM to disk (swap space) to free memory for other processes, and brought back later when needed.

1

Process P1 running in RAM

2

New process P3 needs more RAM

Not enough space

3

Dispatcher swaps out P2 to disk

Swap Out — P2 moves to swap space

4

P3 loaded into freed RAM

Swap In

Concrete Example
RAM = 100 MB
P1 = 40 MB, P2 = 30 MB, P3 = 50 MB

Load P1 + P2 → 70 MB used, 30 MB free
P3 needs 50 MB → Not enough!
Swap out P2 (30 MB) → 60 MB free
Load P3 ✅ — P1 and P3 in RAM, P2 on disk
✅ Pros
  • More processes run simultaneously
  • Better CPU utilization
❌ Cons
  • Disk is very slow vs RAM
  • High overhead if frequent
Key Terms
Dispatcher — module that hands CPU control to the selected process. It saves/restores context, switches to user mode, and jumps to the right instruction.
Swap Out → move process RAM → disk  |  Swap In → move process disk → RAM  |  Swap Space → disk area reserved for swapping

Demand Paging & Page Faults

Definition

Demand Paging = load a page into RAM only when it is actually needed. The OS uses a lazy swapper (pager) — never loads a page unless the CPU requests it. Initially, no pages are in memory.

Page Fault

A page fault occurs when the CPU tries to access a page that is not currently in RAM. The valid/invalid bit in the page table entry is checked — if invalid (i), a page fault trap is triggered and the OS loads the page from disk.

Page Fault Handling — Step by Step

1

CPU tries to access a page

MMU checks page table — valid bit = i (invalid/not in RAM)

2

Trap to Operating System

CPU raises a page fault exception; control goes to OS

3

OS checks if access is valid

Is this page part of the process's address space? If not → segmentation fault / kill process

4

Find a free frame in RAM

If no free frame → run page replacement algorithm to pick a victim

5

Load page from disk into the frame

Disk I/O operation — the slow part

6

Update page table

Set frame number, set valid bit = v

7

Restart the faulting instruction

Execution continues normally

Valid–Invalid Bit in Page Table

Bit ValueMeaningEffect
v (valid = 1)Page is in RAM — accessibleNormal translation proceeds
i (invalid = 0)Page is on disk OR not part of address spacePage fault trap → OS loads the page

Initially all bits are set to i. They change to v as pages are loaded on demand.

07

Page Replacement Algorithms

When Needed?

When a page fault occurs and no free frame is available, the OS must evict (replace) an existing page. The page replacement algorithm decides which page to kick out. Goal: minimize the number of page faults.

Basic Page Replacement Steps
1. Find the required page on disk  →  2. Find a free frame (or pick a victim if none)  →  3. If victim was modified (dirty=1), write it back to disk  →  4. Load required page into frame  →  5. Update page table  →  6. Restart instruction

🥇 FIFO — First In, First Out Simplest

Replaces the page that has been in memory the longest (arrived first). OS keeps a queue — oldest page is at the front.

Example: Reference string: 7, 0, 1, 2, 0, 3, 0, 4  |  Frames = 3

Step70120304
Frame 1 77722224
Frame 2 —0000333
Frame 3 ——111100
Fault? ✗✗✗✗—✗✗✗
⚠️ Belady's Anomaly — FIFO Only!
Normally: more frames = fewer faults. But with FIFO, adding more frames can increase page faults! This paradox is called Belady's Anomaly.
Example: Reference string 3,2,1,0,3,2,4,3,2,1,0,4 → 3 frames = 9 faults; 4 frames = 10 faults. 😱
LRU and Optimal do NOT suffer from Belady's Anomaly.

🥈 LRU — Least Recently Used Practical Best

Replaces the page that has not been used for the longest time. Based on the principle: "if a page was used recently, it'll likely be used again soon."

Key Insight
Uses the Reference Bit in the PTE to track recent access. Good approximation of optimal behavior. Does NOT suffer from Belady's Anomaly.

🥇 Optimal (OPT) Best Possible

Replaces the page that will not be used for the longest time in the future. Produces the minimum possible page faults.

Important Exam Point
Cannot be implemented in practice — you can't predict the future! Used as a benchmark to compare other algorithms. Does NOT suffer from Belady's Anomaly.

Segmentation

Definition

Segmentation is a memory management scheme that divides a process into logical segments of variable size — code, data, stack, heap. Each segment represents a logical unit. Supports the user's view of memory (unlike paging which is purely mechanical).

How Segmentation Works

A logical address is a pair: <segment number, offset>

Physical Address = Base[segment] + Offset
1

CPU generates <segment, offset>

2

Check: offset < limit?

If offset ≥ limit → TRAP (Segmentation Fault)

3

Physical = Base + Offset

Look up segment table for base address

453 > 400 → TRAP! ❌

✅ Segmentation Advantages
  • Natural protection within segments
  • Easy sharing (multiple processes reference same segment)
  • No internal fragmentation
  • Segment tables smaller than full page tables
❌ Segmentation Disadvantages
  • External fragmentation — variable-size holes in memory
  • More costly memory management algorithm
  • Compaction needed to reduce external fragmentation

Thrashing

Definition

Thrashing occurs when the OS spends more time swapping pages in and out of disk than actually executing processes. Page faults happen so frequently that CPU utilization drops dramatically.

How Thrashing Happens — The Vicious Cycle

Key Exam Point
Thrashing = high page fault rate + low CPU utilization. The OS must reduce the degree of multiprogramming (run fewer processes at once) to resolve thrashing.

7 Common Memory Errors — Know These!

  • Forget to allocate — use pointer without malloc → crash
  • Buffer overflow — allocate too little (e.g., forget null char in string)
  • Uninitialized memory — use memory before setting values → garbage data
  • Memory leak — forget to call free() → memory accumulates, system slows
  • Dangling pointer — free memory but keep using the pointer → corruption
  • Double free — call free() twice on same pointer → crash / undefined behavior
  • Invalid free — free a pointer not from malloc() → very dangerous
Managed Languages (Java, Python)
Automatic garbage collection handles deallocation. No need for free(). Memory leaks still possible (forgotten references), but most common errors are avoided.

Quick Revision — Key Formulas & Facts

📐 Key Formulas

Physical Address = Frame# × Page Size + Offset
Page# bits = m − n  |  Offset bits = n
Physical Address (dynamic relocation) = VA + Base
Segmentation PA = Base[seg] + Offset
TLB Hit Rate = Hits / Total Accesses

🧠 Must-Remember Facts

  • Page size = Frame size (always!)
  • Paging eliminates external fragmentation but causes internal
  • Segmentation eliminates internal fragmentation but causes external
  • TLB hit = 1 memory access; miss = 2 memory accesses
  • Only FIFO suffers Belady's Anomaly
  • Optimal = minimum faults, but not implementable
  • Thrashing = high page faults + low CPU utilization
  • Demand paging = lazy loading (only when needed)
  • Dirty bit = 1 → must write to disk before evicting page
